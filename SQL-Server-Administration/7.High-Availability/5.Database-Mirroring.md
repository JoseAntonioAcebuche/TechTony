**Author:** Jose Antonio "Tony" Acebuche"

**Category:** SQL Server High Availability

**Purpose:** Learn how SQL Server Database Mirroring works, understand its architecture, operating modes, failover process, monitoring, and why it has been replaced by Always On Availability Groups.

---

# 📖 Overview

**Database Mirroring** is a legacy High Availability (HA) solution introduced in SQL Server 2005.

It maintains two synchronized copies of a database:

- Principal Database
- Mirror Database

Every transaction committed on the Principal database is immediately sent to the Mirror database.

Although Database Mirroring is **deprecated**, many organizations still use it in legacy SQL Server environments.

Microsoft recommends using **Always On Availability Groups** for new deployments.

---

# Why Use Database Mirroring?

Benefits include:

- High Availability
- Automatic failover (with Witness)
- Near real-time synchronization
- Fast recovery
- Simple architecture
- Database-level protection

# Operating Modes

| Mode | Automatic Failover | Performance |
|------|--------------------|-------------|
| High Safety | No | Synchronous |
| High Safety with Witness | Yes | Synchronous |
| High Performance | No | Asynchronous |

---

# High Safety Mode

Characteristics:

- Synchronous replication
- Zero or minimal data loss
- Manual failover
- No Witness required

```text
Principal

⇄

Mirror
```

---

# High Safety with Witness

Characteristics:

- Synchronous replication
- Automatic failover
- Witness server required
- High availability

```text
Principal

⇄

Mirror

      ▲

      │

  Witness
```

---

# High Performance Mode

Characteristics:

- Asynchronous replication
- Better performance
- Possible data loss
- Manual failover

```text
Principal

↓

Mirror
```

# Prerequisites

Before configuring Database Mirroring:

- SQL Server Standard or Enterprise Edition
- FULL Recovery Model
- Full Database Backup
- Transaction Log Backup
- Database restored with NORECOVERY
- Reliable network connection
- Mirroring endpoints configured

---

# Recovery Model

Verify recovery model:

```sql
SELECT

name,

recovery_model_desc

FROM sys.databases;
```

Enable FULL Recovery:

```sql
ALTER DATABASE ParkingDB

SET RECOVERY FULL;
```

# Configure Database Mirroring

```text
Create Full Backup

↓

Restore Database (NORECOVERY)

↓

Create Mirroring Endpoint

↓

Configure Principal

↓

Configure Mirror

↓

Configure Witness (Optional)

↓

Start Mirroring
```

---

# Perform Manual Failover

```sql
ALTER DATABASE ParkingDB

SET PARTNER FAILOVER;
```

---

# Remove Mirroring

```sql
ALTER DATABASE ParkingDB

SET PARTNER OFF;
```

# Automatic Failover

Automatic failover requires:

- High Safety Mode
- Witness Server
- Healthy network connection
- Synchronized databases

---

# Advantages

- Automatic failover
- Near real-time synchronization
- Simple setup
- High availability
- Database-level protection

---

# Limitations

- Deprecated feature
- Only protects one database at a time
- No readable mirror database
- Limited scalability
- Replaced by Always On Availability Groups

---

# Best Practices

- Use FULL Recovery Model.
- Regularly monitor mirroring status.
- Test failover periodically.
- Back up the Principal database.
- Monitor endpoint connectivity.
- Plan migration to Always On AG.

---

# Common Mistakes

❌ Forgetting FULL Recovery Model.

❌ Not restoring with NORECOVERY.

❌ Ignoring endpoint configuration.

❌ Never testing failover.

❌ Deploying Database Mirroring for new projects.

# Notes

> **Database Mirroring is a deprecated feature.**

> Microsoft recommends using **Always On Availability Groups** for all new SQL Server deployments. Database Mirroring remains relevant only for maintaining or supporting legacy SQL Server environments.
