# 📖 Overview

**Always On Availability Groups (AG)** is Microsoft's modern High Availability and Disaster Recovery (HADR) solution introduced in SQL Server 2012.

It protects one or more user databases by replicating them to one or more secondary replicas.

Unlike Log Shipping, Always On supports:

- Automatic Failover
- Multiple Secondary Replicas
- Readable Secondary Databases
- Automatic Data Synchronization
- Backup Offloading

Always On AG is considered the recommended HA solution for modern SQL Server deployments.

# Prerequisites

Before configuring Always On AG:

- Windows Server Failover Cluster (WSFC)
- SQL Server Enterprise Edition (recommended)
- Databases using FULL Recovery Model
- Database Full Backup
- Transaction Log Backup
- Reliable network connection
- SQL Server service account permissions

---

# Recovery Model

Verify the recovery model:

```sql
SELECT

name,

recovery_model_desc

FROM sys.databases;
```

Enable Full Recovery:

```sql
ALTER DATABASE ParkingDB

SET RECOVERY FULL;
```

---

# Synchronization Modes

| Mode | Description |
|------|-------------|
| Synchronous Commit | Zero or near-zero data loss |
| Asynchronous Commit | Better performance over long distances |

---

# Failover Modes

| Mode | Description |
|------|-------------|
| Automatic Failover | Supported with synchronous replicas |
| Manual Failover | Administrator initiates failover |
| Forced Failover | Disaster recovery scenario |

# Readable Secondary

Secondary replicas may be configured for:

- Reporting
- Read-only queries
- Backup operations
- Disaster recovery

This reduces workload on the production server.

---

# Availability Listener

The Availability Listener provides a single virtual hostname for client connections.

Instead of connecting directly to a SQL Server instance:

```text
Application

↓

SQLListener.company.local

↓

Current Primary Replica
```

Applications continue using the same connection string even after failover.

# Automatic Failover

Automatic failover requires:

- Two synchronous replicas
- WSFC healthy
- Automatic failover enabled
- Both replicas synchronized

---

# Backup Preferences

Always On allows backups on:

- Primary Replica
- Preferred Secondary Replica
- Any Replica

This reduces backup load on production.

---

# Advantages

- Automatic failover
- Readable secondary replicas
- Backup offloading
- High availability
- Disaster recovery
- Near real-time synchronization
- Enterprise scalability

---

# Limitations

- Requires Windows Server Failover Cluster
- More complex configuration
- Additional licensing costs
- Higher hardware requirements
- User databases only (system databases are excluded)

---

# Best Practices

- Use synchronous commit for local HA.
- Use asynchronous commit for remote DR sites.
- Regularly test failover.
- Monitor synchronization health.
- Configure backup preferences.
- Use Availability Listener for all applications.
- Monitor replica latency.

---

# Common Mistakes

❌ Forgetting FULL Recovery Model.

❌ Ignoring WSFC health.

❌ Not testing failover.

❌ Connecting applications directly to SQL Server instead of the Listener.

❌ Ignoring synchronization warnings.
