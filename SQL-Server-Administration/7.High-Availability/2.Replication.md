# 📖 Overview

**SQL Server Replication** is a technology used to copy and distribute data from one database to another while maintaining data consistency.

Unlike Log Shipping, Replication is designed for **data distribution**, allowing selected tables or objects to be synchronized between multiple servers.

Replication is commonly used for:

- Reporting databases
- Remote office synchronization
- Data warehousing
- Load balancing
- Data integration

---

# Why Use Replication?

Benefits include:

- Near real-time data synchronization
- Reduced reporting load on production servers
- Data distribution across multiple locations
- Supports multiple subscribers
- Improved scalability
- Minimal application downtime

# Replication Components

| Component | Purpose |
|-----------|---------|
| Publisher | Source database |
| Distributor | Stores replication metadata |
| Subscriber | Receives replicated data |
| Publication | Collection of replicated objects |
| Subscription | Connection between Publisher and Subscriber |
| Article | Individual database object |

---

# Replication Types

| Type | Description |
|------|-------------|
| Snapshot Replication | Copies entire dataset |
| Transactional Replication | Continuously replicates transactions |
| Merge Replication | Synchronizes changes from multiple servers |
| Peer-to-Peer Replication | Enterprise-scale replication between servers |

---

# Snapshot Replication

Creates a complete copy of published data.

```text
Publisher

↓

Snapshot

↓

Subscriber
```

Best for:

- Small databases
- Infrequent data changes
- Initial synchronization

---

# Transactional Replication

Continuously sends committed transactions.

```text
INSERT

↓

Transaction Log

↓

Distributor

↓

Subscriber
```

Best for:

- Reporting databases
- High transaction systems
- Near real-time replication

---

# Merge Replication

Allows both Publisher and Subscriber to update data.

```text
Publisher

⇅

Subscriber
```

Conflict detection and resolution are built into Merge Replication.

Best for:

- Mobile users
- Branch offices
- Occasionally connected systems

---

# Peer-to-Peer Replication

Every server acts as both Publisher and Subscriber.

```text
Server A

⇄

Server B

⇄

Server C
```

Designed for enterprise environments with high availability.

# Advantages

- Near real-time synchronization
- Multiple subscribers
- Supports reporting servers
- Reduces production workload
- Selective table replication

---

# Limitations

- More complex than Log Shipping
- Requires additional administration
- Conflict management for Merge Replication
- Replication latency may occur
- Not a complete disaster recovery solution

---

# Best Practices

- Use Transactional Replication for reporting.
- Monitor replication latency.
- Keep network connections reliable.
- Regularly back up Publisher and Distributor.
- Limit replicated objects to only what is necessary.
- Monitor SQL Server Agent jobs.

---

# Common Mistakes

❌ Replicating unnecessary tables.

❌ Ignoring replication latency.

❌ Forgetting Distributor backups.

❌ Poor network configuration.

❌ Not monitoring replication agents.
