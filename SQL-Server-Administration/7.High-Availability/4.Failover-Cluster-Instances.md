# 📖 Overview

**SQL Server Failover Cluster Instance (FCI)** is a High Availability (HA) solution that protects an entire SQL Server instance by using multiple Windows servers (nodes) connected to shared storage.

Only one node actively runs the SQL Server instance at a time. If the active node fails, another node automatically takes ownership of the SQL Server instance with minimal downtime.

Unlike Always On Availability Groups, FCI protects the **entire SQL Server instance**, including:

- User Databases
- System Databases
- SQL Server Logins
- SQL Server Agent Jobs

FCI is designed for **High Availability**, not Disaster Recovery.

# Why Use Failover Cluster Instances?

Benefits include:

- Automatic failover
- Protects the entire SQL Server instance
- Minimal downtime
- Shared storage architecture
- Transparent client connectivity
- Automatic recovery after hardware failure

# Prerequisites

Before configuring FCI:

- Windows Server Failover Cluster installed
- SQL Server Enterprise or Standard Edition
- Shared Storage (SAN, iSCSI, Fibre Channel)
- Domain-joined servers
- Static IP addresses
- Matching SQL Server versions
- Reliable network connectivity

```text

      ▼
```

# Automatic Failover

Occurs when:

- Active node crashes
- Hardware failure
- Operating system failure
- SQL Server service failure

WSFC automatically transfers the SQL Server role to another node.

---

# Advantages

- Automatic failover
- Protects the complete SQL Server instance
- Minimal downtime
- Transparent to applications
- System databases are protected
- SQL Agent jobs remain available

---

# Limitations

- Requires shared storage
- Does not protect against storage failure
- More expensive infrastructure
- Not a Disaster Recovery solution
- Requires Windows Server Failover Cluster

---

# Best Practices

- Use redundant SAN storage.
- Validate the cluster before deployment.
- Regularly test failover.
- Monitor cluster events.
- Keep all nodes patched consistently.
- Use dedicated cluster networks.
- Back up databases regularly.

---

# Common Mistakes

❌ Skipping cluster validation.

❌ Single point of failure in storage.

❌ Ignoring Windows Event Logs.

❌ Different SQL Server patch levels between nodes.

❌ Never testing failover.
