# 📖 Overview

**Log Shipping** is a High Availability (HA) and Disaster Recovery (DR) feature in SQL Server.

It automatically copies **Transaction Log Backups** from a Primary Server to one or more Secondary Servers.

The Secondary Server continuously restores these transaction logs, keeping a standby copy of the database nearly synchronized with the Primary Server.

Log Shipping is designed primarily for **Disaster Recovery**, not automatic failover.

---

# Why Use Log Shipping?

Benefits include:

- Disaster Recovery
- Secondary standby database
- Low implementation cost
- Easy to configure
- Supports multiple secondary servers
- Simple maintenance
- Backup redundancy

---

# Log Shipping Architecture

```text
                 Primary Server

                Production Database

                        │

      Transaction Log Backup Job

                        │

                        ▼

               Backup Share (Network)

                        │

                 Copy Log Backup Job

                        ▼

                Secondary Server

                 Restore Log Job

                        ▼

            Standby / Read-Only Database
```

| Restore Job | Restores transaction logs |

# Requirements

Before configuring Log Shipping:

- SQL Server Enterprise or Standard Edition
- Full Recovery Model
- Shared network folder
- SQL Server Agent running
- Initial Full Backup restored on Secondary
- Reliable network connection

---

# Recovery Model

Log Shipping requires the database to use the **FULL Recovery Model**.

Check recovery model:

```sql
SELECT

name,

recovery_model_desc

FROM sys.databases;
```

Enable Full Recovery:

```sql
ALTER DATABASE ParkingDB

SET RECOVERY FULL;
```

# Backup Job

Runs on the Primary Server.

Responsibilities:

- Create transaction log backups
- Store backup files
- Maintain backup history

Example Schedule:

```
Every 15 Minutes
```

---

# Copy Job

Runs on the Secondary Server.

Responsibilities:

- Copy transaction log backups
- Verify copied files
- Retry failed copies

---

# Restore Job

Runs on the Secondary Server.

Responsibilities:

- Restore transaction logs
- Keep secondary synchronized
- Maintain standby database

Restore Modes:

- No Recovery
- Standby (Read-Only)

---

# Standby Mode

Standby mode allows:

- Read-only reporting
- Backup verification
- Disaster Recovery testing

Users may query the secondary database while waiting for the next restore cycle.

# Failover Process

Log Shipping does **NOT** provide automatic failover.

Manual Failover Steps:

1. Stop applications.
2. Back up the final transaction log (Tail Log Backup).
3. Copy the final backup.
4. Restore remaining logs.
5. Recover the secondary database.
6. Redirect applications.

---

# Tail Log Backup

Before failover:

```sql
BACKUP LOG ParkingDB

TO DISK='D:\Backup\ParkingDB_Tail.trn'

WITH NORECOVERY;
```

---

# Recover Secondary Database

```sql
RESTORE DATABASE ParkingDB

WITH RECOVERY;
```

After recovery, the secondary database becomes writable.

# Advantages

- Easy to configure
- Supports multiple standby servers
- Low hardware cost
- Excellent Disaster Recovery solution
- Read-only reporting server
- Independent backups

---

# Limitations

- Manual failover
- Possible data loss between log backups
- Secondary database is not always current
- Requires SQL Server Agent
- Dependent on network connectivity

---

# Common Failure Scenarios

| Issue | Possible Cause |
|--------|----------------|
| Backup Job Failed | SQL Agent stopped |
| Copy Job Failed | Network issue |
| Restore Failed | Missing transaction log |
| Secondary Behind | Restore schedule too slow |
| Monitor Alerts | Backup threshold exceeded |

---

# Best Practices

- Schedule log backups every 5–15 minutes.
- Monitor SQL Server Agent jobs.
- Store backups on reliable storage.
- Test failover regularly.
- Monitor available disk space.
- Verify backup integrity.
- Keep server clocks synchronized.

---

# Common Mistakes

❌ Using SIMPLE Recovery Model.

❌ Forgetting SQL Server Agent.

❌ Never testing failover.

❌ Deleting transaction log backups too early.

❌ Ignoring monitoring alerts.
